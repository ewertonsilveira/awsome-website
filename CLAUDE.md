# Project Context for AI Agents

## About This Project

**AWSome Painting & Decorating — Marketing Site** (`SPEC-2026-01`).

A marketing website for a Wellington-region painting and decorating business. Built with React 19 + TanStack Start + Tailwind v4, deployed to **Cloudflare Pages**. The pages are statically prerendered; a single Cloudflare Pages Function (`web-app/functions/api/contact.ts`) backs the contact form (email via Cloudflare Email Service + KV-backed rate limiting). All application code lives in `web-app/`.

```

                 server-side code is the Cloudflare Pages Function under functions/

Styling:         Tailwind CSS v4 (@tailwindcss/vite plugin)

                 Pages Function (functions/api/contact.ts re-validates every request)
Contact form:    Cloudflare Pages Function — ContactForm POSTs JSON to /api/contact;
                 functions/api/contact.ts validates (Zod), rate-limits (KV), emails (Cloudflare Email Service REST API)
Email:           Cloudflare Email Service REST API (functions/api/contact.ts) — requires `CF_EMAIL_API_TOKEN` secret + `CF_ACCOUNT_ID`
Edge state:      Cloudflare KV namespace RATE_LIMIT (binding in wrangler.toml) — per-IP rate limit
Security headers: web-app/public/_headers (Cloudflare Pages convention) — CSP + HSTS + frame/MIME guards
Lint/format:     Prettier + tsc --noEmit (no ESLint / Vitest / husky in v1)
Package manager: npm — standardised; never use pnpm or yarn in this repo
Runtime:         Node.js v24.3.0 (build); Cloudflare Workers runtime for the Pages Function
Hosting/CD:      Cloudflare Pages (project "awsome-nz"); deploy via `npm run deploy:cf` (wrangler)
App code lives:  web-app/
Awsome Prod URL: https://www.awsome.co.nz
Hosted on:       https://www.godaddy.com (domain registrar; site served by Cloudflare Pages)
```

│   ├── data/             # Hard-coded business content as typed constants
│   │   └── *.ts          # business.ts, services.ts, nav.ts, content.ts, contact.ts (Zod schema)

├── functions/            # Cloudflare Pages Functions (server-side, Workers runtime)
│   └── api/contact.ts    # POST /api/contact — Zod validate + KV rate-limit + Cloudflare Email Service
├── public/               # Static assets served at root (images, favicon)
│   └── _headers          # Cloudflare Pages security headers + CSP
├── vite.config.ts        # Vite + TanStack Start config (prerender pages, plugins)
├── wrangler.toml         # Cloudflare Pages config (output dir, KV binding, compat date)
├── tsconfig.json         # TypeScript strict config (strict: true, paths: ~/→src/)
└── package.json          # npm scripts: dev, build, format:check, preview, deploy:cf

**If unsure where new code belongs:**
- New page? → `src/routes/<name>.tsx`
- Reusable UI piece (no data fetching)? → `src/components/<Name>.tsx`
- Business content / copy? → `src/data/<domain>.ts`
- One-off utility pure function? → `src/utils/<name>.ts` (create if needed)

## Conventions

- **Naming**: PascalCase for React components and their files (`ServiceCard.tsx`). camelCase for everything else (hooks, utils, data, constants). SCREAMING_SNAKE_CASE for true module-level constants.
- **Imports**: Always use the `~/` alias (`~/components/Button`, `~/data/services`). Never use deep relative paths (`../../../`). `routeTree.gen.ts` is the only file imported by alias AND committed.
- **Components**: Functional only — no class components. Keep components presentational: props in, events/callbacks out. Page-level routes own layout/content composition; `src/components/` holds reusable pieces.
- **Data**: Business content (copy, lists, contact info) lives in `src/data/` as `export const` typed values — not hard-coded inside JSX.
- **Async**: `async/await` only. No `.then()` chains. All async is in route loaders or event handlers — not in component bodies.
- **Error handling**: Typed errors. Never swallow silently. Form validation errors surface via accessible `aria-describedby` + `aria-invalid` attributes.
- **TypeScript**: `strict: true` is non-negotiable. No `any`, no `@ts-ignore`, no `as unknown as X` escape hatches. Zod `z.infer<>` for all form value types.
- **Styling**: Tailwind utility classes only — no inline `style=` for layout/colour. Theme tokens defined in `app.css`; extend there, not in component files.
- **No SSR in pages**: Routes/components are statically prerendered — no `useServerFn`, no server-only imports, no `process.env` reads at render time. The ONLY server-side code is `functions/api/contact.ts`; keep all backend logic (secrets, KV, Cloudflare Email Service, `env.*` bindings) there, never in `src/`.
- **Server-side validation**: The Pages Function re-validates every request with Zod — never trust the client. Client-side Zod (`src/data/contact.ts`) is for UX only; the function owns the security boundary.

## Testing Rules (v1 — Lean)

No automated unit/integration/E2E test suite in v1 (explicit decision in SPEC-2026-01). Quality is enforced by:

1. **Build gate** (primary): `cd web-app && npm run build` MUST exit 0. This runs Vite build + `tsc --noEmit` and is the hard quality gate.
2. **Format gate**: `cd web-app && npm run format:check` (Prettier) MUST pass with no diffs.
3. **Manual verification**: dev server, all four routes (`/`, `/about`, `/contact`, a 404), and a contact-form submission against the deployed Pages Function (`/api/contact`) on a Cloudflare preview deployment — the form's email + rate-limit path can only be verified end-to-end on a deploy, not at build time.

Agents run these gates after every meaningful change: `npm run build && npm run format:check` from `web-app/`.

Adding Vitest + Testing Library is explicitly deferred to a follow-on spec. Do not add test infrastructure in v1.

## Do Not Touch Without Explicit Approval

These paths are protected by hooks — agents cannot write to them without a human override:

- `migrations/**` — database schema changes need human review.
- `infra/**`, `terraform/**`, `.github/workflows/**` — infrastructure needs human review.
- `.env*` files — never read, never write. Use the secrets manager.
- Package manifest dependencies (`package.json`, `pyproject.toml`, `go.mod`, etc.) — propose additions in chat, do not install silently.

> Adjust `.claude/hooks/block-protected-paths.mjs` to match your actual protected paths when branching.

## Agent Operating Rules

These rules apply to ALL agents in this repo, regardless of stack:

1. **Spec first**: For any task larger than ~50 LOC or touching >2 files, write/update a spec under `specs/` before coding (see `.claude/commands/spec.md`).
2. **Three-strikes rule**: If a test or build fails 3 times on the same root cause, stop. Summarize what you tried, your current hypothesis, and escalate to the human.
3. **Plan before edit**: Use the `architect` subagent for cross-file changes. Confirm the plan with the human before touching code.
4. **Peer review every task**: After each implementation task — and after its gates pass but **before it is committed** — the `reviewer` agent runs an adversarial pass on the task diff (the same creator/validator pattern the `plan-reviewer` applies to designs). Fold Critical/Important findings back via the `coder`; the reviewer never fixes its own findings. Do not skip this step.
5. **Verify before claiming done**: After every meaningful change run the gates from `web-app/` — `npm run build` (vite build + `tsc --noEmit`) and `npm run format:check` — **and run the app** (dev server / preview) to confirm it actually works, not just that it compiles. Add/run tests only where the task or design calls for them (v1 has no automated suite — see Testing Rules). If any gate fails, do not declare success.
6. **Diff hygiene**: Make the smallest diff that satisfies the spec. No unrelated reformatting. No "while I'm here" cleanups.
7. **One clean commit per task**: A `/implement` task is done only when the gates pass, the app has been run, and the reviewer has approved — then commit it as a single Conventional Commit (`feat:`, `fix:`, `refactor:`, `test:`, `chore:`) referencing the spec ID in the body. Each task leaves a clean working tree so the next `/implement` starts fresh.
8. **No new dependencies** without justifying it in the PR description (size, maintenance status, alternatives considered).
